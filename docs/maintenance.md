# 制度維護協議（通用版）

制度只有被維護才有價值，但錯誤的維護比不維護更糟。本檔定義任何 agent 更新本套組
（或其在專案內的副本）時的安全邊界。

## 權限分級

**可自行改（改完回報一行即可）**
- 環境事實檔：新增你**親自實測**過的事實、修正過時項。
- 錯字、失效路徑、壞連結（修正前先驗證新路徑存在）。
- 為既有判準補充正/反例（本次工作出現教科書級案例時）；不可改判準本文。

**動之前必須先問人類**
- 入口檔的硬規則區與規則優先序。
- model-dispatch 的升降級路徑與「驗證不自驗」（弱化它們的修改往往偽裝成「效率優化」）。
- 刪除任何檔案或整個章節。

**嚴禁**
- 因為「這條規則這次不方便」而當場改規則——不方便就記錄下來讓人類裁決。
- 用追加例外架空規則（「原則上 X，但以下 N 種情況除外」）。

## 教訓寫回對照表（問題確認的當下就寫，別等收尾）

| 教訓類型 | 寫到 | 格式 |
| --- | --- | --- |
| 環境/工具客觀事實 | 專案環境檔 | 一行事實＋一行正確作法 |
| 判斷失誤（假完成、該升級沒升級） | 對應 skill 的判準下加反例 | ❌ 一行情境＋一句為什麼錯 |
| 流程瓶頸、制度修改提案 | 專案的檢討/提案目錄 | 現象→成因→提案 |
| 程式 bug 成因與解法 | 專案問題日誌 | 現象→根因→解法→驗證 |

## 膨脹控制

- 入口檔 ≤80 行；單一 SKILL.md ≤120 行；超標的正確動作是**精簡**（合併重複、刪過時），不是開新檔。
- 每條判準的例子最多 2 正 2 反，加新例就淘汰最弱的舊例。
- 同一條規則只准有一個活的原文位置；發現第二處在「複述」而非「引用」，就地改成指針。

## 已知退化模式與預防（按發生機率排序）

1. **例外累積架空核心規則**：預防——例外不寫回規則；擴充例外清單必走「先問人類」通道。
2. **環境檔漂移後被整體不信任**：預防——發現一條失效當下就修那一條（免請示），一條不修比修錯更糟。
3. **驗證儀式化**（fresh-context 驗收退化成橡皮圖章）：預防——守住兩條紅線：驗收 prompt
   不夾帶實作者結論、問「找出問題」不問「確認沒問題」。
4. **文件膨脹到沒人讀**：預防——行數上限是硬的。

## 上游 ↔ 專案 更新通道

- **上游 → 專案**：在專案裡重跑 `node <套組repo>/setup.mjs`。lockfile 三方規則：
  目標檔 hash 與 lockfile 一致（沒被本地改過）→ 安全升級到新版；不一致（本地寫回過教訓）→
  保留本地版並警告，由人決定手動合併或 `--force`。
- **專案 → 上游**：專案內累積的通用教訓（不含專案專屬事實）以 PR/issue 回流套組 repo，
  格式照 CONTRIBUTING.md；專案專屬的教訓留在專案的環境檔與問題日誌，不上行。
- 判斷通用還是專屬：換一個技術棧完全不同的專案，這條教訓還成立嗎？成立才上行。

## 死 skill 防治

技能清單上每一格都是每 session 的固定 token 成本。內容宣告「一次性、已完成、永不再跑」
的 skill 必須退役（移出技能目錄、留備份）；同一能力不得以多個來源同時列在技能清單
（例如 plugin 版與專案副本並存）。
