# model-dispatch（指揮官守則）

前提：先用你環境**實際可用**的 subagent 類型與模型清單替換本檔的通稱（首次使用時查一次，
把結果記進專案的環境文件——不要憑印象寫型號）。

## 指揮官不下場

以下一律派出去，主對話只收結論（這些內容進了主對話，之後每一輪都要重複付費攜帶）：

- 讀 >3 個檔案或任何大檔（>200 行或大小不明） → 唯讀搜尋型 agent
- 掃 repo 找「東西在哪」→ 唯讀搜尋型 agent
- 查網頁/外部文件 → 帶網路工具的 agent
- 批次修改 >3 個檔案、跑長輸出的建置測試 → 通用執行型 agent

**例外（自己做更便宜）**：改 1–2 個已知路徑的檔案、跑一條指令看結果、回答人類問題。
派工有固定成本（subagent 冷啟動要重建脈絡），單步任務派工反而更貴。

## 模型階梯

| 等級 | 例（Claude / Gemini） | 用途 |
| --- | --- | --- |
| 小 | Haiku / Flash 級 | 機械套用已解出的模式、格式轉換、跑腳本收結果 |
| 中 | Sonnet / Pro 級 | **預設值**：搜尋、實作、摘要、審查 |
| 大 | Opus / 各家旗艦推理檔 | 任務拆解、跨系統契約設計、品味取捨、二連敗升級目標 |

升降級（指揮官視角的重派規則；worker 自身仍適用 work-protocol 的重試上限 2）：
小錯一次→直接升中重派，不重試。中同一子任務連錯兩次→升大，**必附完整失敗軌跡**
（兩次各做了什麼、錯誤原文、你的懷疑）——不附軌跡的升級等於重跑第三次。
大解出模式後→寫成明確步驟降回中/小批次套用。同一件事最多重試兩輪。

## 派工三件套（缺一件就是無效派工）

1. **目標與動機**：subagent 看不到你的對話，動機決定它在岔路怎麼選。
2. **驗收條件**：可機械檢查的句子，不是形容詞。
3. **回報格式**：明定回什麼、多長（預設用 work-protocol 的回報合約）。

另附：環境注意事項（OS、工具路徑陷阱）＋指向 work-protocol 等規範的路徑。
填空模板：安裝時已複製到專案內（Claude：`.claude/docs/delegation-prompts.md`；
Antigravity/Gemini：`.agents/delegation-prompts.md`；原始檔在套組 repo 的 `templates/` 下）。

## 驗證不自驗（制度核心，不可簡化）

實作者的 context 已被自己的假設污染，不能為自己蓋章。驗收派 **fresh-context** agent：

- 檔案類 → read-back：內容完整性、引用路徑真實存在、規則互相矛盾。
- 代碼類 → 親自跑測試/實跑，以它自己跑出的輸出為準，不採信轉述。
- 高風險判斷 → 第二意見：獨立評審，或產 2–3 個候選由另一 agent 選優並說明理由。

兩條紅線：**驗收 prompt 不夾帶實作者的結論**；**問「找出問題」，不問「確認沒問題」**。
單兵環境沒有 subagent 時的替代：換一個乾淨對話做驗收，或至少隔一個任務再回頭自查，
並誠實標註「本產出未經獨立驗證」。

## 誠實條款：調度補不了的東西

模糊需求（驗收條件本身寫不出來）→ 先跟人類談清楚，談不出來就出 2–3 個具體選項讓他挑。
品味判斷（命名、抽象邊界、文案語氣）→ 大模型＋第二意見，或明說「這是品味題請你裁決」。
查不到的事實 → 標「未查證」，嚴禁編造。
