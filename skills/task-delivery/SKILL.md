# task-delivery（產出類任務）

## 文件（docs）

**先問這份文件的讀者是誰、讀完要能做什麼**——答不出來就不要寫。固定三型：

| 型 | 結構 | 判斷寫得好不好 |
| --- | --- | --- |
| 使用/操作型 | 前置需求 → 步驟（每步含預期結果）→ 排錯表 | 新人照著做能一次成功 |
| 架構/設計型 | 目標 → 圖 → 分模組職責 → 關鍵決策與取捨（為什麼不用 X） | 接手者不用讀代碼就知道東西放哪、為什麼 |
| 決策紀錄型（ADR） | 背景 → 選項 → 決定 → 後果；一件事一份，追加不改寫 | 半年後能回答「當初為什麼這樣做」 |

紀律：文件中的每個路徑/指令寫完當場驗證一次；過時文件比沒文件危險，改代碼動到文件描述的行為時，同一批 commit 更新文件。

## 架構圖（mermaid，進版控）

- 模組關係用 `graph TD`、時序用 `sequenceDiagram`、狀態機用 `stateDiagram-v2`。
- 一張圖一個主題，節點 ≤12 個；超過就拆層級（總覽圖＋子系統圖）。
- 圖跟著代碼走：放在對應 docs 的 markdown 內，不要另存孤兒圖檔。
- 節點名用代碼裡的真實名稱（類別/模組名），不用抽象暱稱——圖是導航工具不是裝飾。

## 版本控管

- **分支**：保護分支永不直接提交（`main`/`master` 是通例；專案可加列，例如有些專案另設 `test` 分支）；特性分支命名 `feat/<主題>`、`fix/<主題>`。
- **Commit**：conventional 格式（`feat:` `fix:` `refactor:` `docs:` `chore:`），一個 commit 一個意圖；選擇性 add（指定路徑），嚴禁無腦 `git add .` 吞掉別人的變更。
- **版本號**：SemVer——破壞相容 +Major、新功能 +Minor、修復 +Patch。
- **CHANGELOG**：每版一節，分 Added/Changed/Fixed/Removed，寫「對使用者的影響」不是 commit 清單的複製貼上。

## 交付物打包（任務收尾的固定格式）

```
【交付清單】每項：路徑 + 一句它是什麼
【驗收核對】對照原始需求逐條：通過/未通過/超出範圍未做（附原因）
【已知限制】沒做完/做不到的部分，明說
【下一步建議】≤3 條，可選
```

交付前 read-back：把每個交付檔案重新打開確認內容完整（不是憑記憶說寫完了）；
檔案間的互相引用（連結、路徑）逐個點過。
