# task-planning（計畫類任務）

產出物是一份**能被別的（更便宜的）agent 直接執行**的計畫書。判斷標準：一個沒讀過
本對話的 agent 拿到計畫書就能開工，不需要回來問你「這是什麼意思」。

## 計畫書固定結構（每節都要有，寫不出來就是還沒想清楚）

```markdown
# <功能名>
## 目標與動機        # 要什麼效果、為什麼現在做、不做會怎樣
## 範圍              # 做什麼（In）/ 明確不做什麼（Out）——Out 至少列 2 條
## 方案              # 架構決策 + mermaid 圖（模組關係或資料流，擇一必畫）
## 分步實作清單      # 每步：做什麼 + 動哪些檔案 + 這步的完成證據是什麼
## 驗收條件          # 可機械檢查的句子（見下方寫法）
## 風險與回退        # 前 3 大風險各附偵測訊號與應對；整體回退方案一句
## 待裁決問題        # Q1..Qn 附選項與預設值（human-intent 的批次提問格式）
```

## 驗收條件寫法（計畫的靈魂，寫爛整份作廢）

- ✅ 可機械檢查：「`POST /api/notes` 對 10MB body 回 413」「新增 20 筆後列表首屏 <200ms」
- ❌ 形容詞：「效能良好」「使用者體驗流暢」「代碼可維護」
- 每條驗收條件必須能回答：「**誰**用**什麼方法**檢查，看到**什麼**算過？」
- 至少一條負面路徑（錯誤輸入/邊界條件該發生什麼）。

## 規劃紀律

1. **嚴格審視原始想法**：人類的提案裡通常混著目標與猜想（human-intent）。發現邏輯漏洞、
   更省的替代方案、被低估的風險——寫進計畫書明講，不盲從。這是規劃者的職責不是冒犯。
2. **拆步的粒度**：每步能在一個工作段內完成並可獨立驗證；一步失敗不應污染其他步。
3. **凍結區標記**：計畫要明列「不准動的既有部分」，防止實作 agent 好心重構到別人的地盤。
4. **技術選型要查證**：計畫中引用的庫/API 必須當場查證存在且版本相容（用文件查詢工具），
   把「預期用法簽名」寫進計畫。規劃期的幻覺會被實作期放大十倍。

## 核准閘門（不可跳過）

計畫書寫完 → 交人類核准 → **拿到明確同意才進開發**。本閘門凌駕 human-intent 的
「已讀不回照預設進行」規則：核准這件事沒有預設值，沉默≠同意。人類提修改就改計畫書再送，
不要「邊改邊做」。核准後計畫書就是合約：實作偏離計畫要回來改計畫書，不是默默偏離。
