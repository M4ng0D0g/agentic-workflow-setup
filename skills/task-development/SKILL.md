# task-development（開發類任務）

## 分層架構（畫面 / 狀態 / 領域資料 / 網路基礎設施）

任何 UI 應用照這四層放代碼；純後端去掉畫面層同理。**依賴只准往下，禁止跳層與回頭**：

| 層 | 放什麼 | 絕不放什麼 |
| --- | --- | --- |
| 畫面（View） | 版面、樣式、把使用者操作轉成事件 | 業務規則、直接呼叫網路/儲存 |
| 狀態（State） | 畫面狀態機、把領域資料轉成畫面要的形狀 | 樣式常數、SQL/HTTP 細節 |
| 領域資料（Domain/Data） | 業務規則、模型、Repository 介面與實作 | 對特定 UI 框架的 import |
| 網路/基礎設施（Infra） | HTTP client、DB driver、檔案系統——全部包在 Wrapper 後面 | 業務判斷 |

判定歸屬的問題：「換掉 UI 框架這段代碼要不要改？」要改→畫面/狀態層；
「換掉 HTTP 庫要不要改？」要改→只准 Infra 層要改。

## 設計模式統一表（同一問題全專案只用同一招）

| 問題 | 用 | 不要 |
| --- | --- | --- |
| 資料存取 | Repository（介面在領域層、實作在 Infra） | 業務代碼直接摸 DB/API |
| 外部庫接入 | Adapter/Wrapper 一層包住 | 第三方型別滲透到領域層 |
| 跨模組通知 | 事件匯流排或觀察者，擇一全專案統一 | 兩者混用、隨手互相 import |
| 物件建構複雜 | Factory | 到處散落的 new + 參數猜謎 |
| 全域狀態 | 單一注入點（DI/Composition Root） | 隨處可改的全域變數 |

新增第二種做法前必須先回答：「既有那招為什麼不能用？」答不出來就用既有的。
**持續生長的註冊點（路由表、DI 綁定）永遠放在獨立的 Bootstrap/Composition Root，
不要放進會被凍結或很少改的入口函式**（歷史教訓：路由掛在 main() 裡，main 被凍結後每個新端點都撞牆）。

## 代碼紀律

- 單檔 ≤200 實質行（實質行＝扣除空行與純註解行；**200 是預設值，專案入口檔可覆寫**）、
  一個 public 類別；逼近上限就拆檔（組合優於繼承）。
- 外部輸入（API body、檔案、環境變數）進門就驗證；錯誤要嘛處理要嘛往上拋，禁止空 catch。
- 關鍵分歧點與 I/O 邊界埋結構化日誌（模組標籤＋上下文）；不留臨時 print。
- 硬編碼禁令：色值、URL、魔術數字收攏到常數/設定/主題層。
- **測試與代碼同批**：新邏輯沒有對應測試＝這批工作沒完成。改跨端契約（前後端、跨語言）時，
  契約先行：先改契約定義與兩端註解，再改實作，最後跑整合測試。

## 修 bug 專用順序

1. 重現（拿到穩定重現步驟才算開始）→ 2. 翻專案問題日誌（踩過沒）→ 3. 定位根因
（錯誤數量爆炸時先分群找根因，不逐個修表象）→ 4. 修＋加回歸測試 → 5. 記錄
（成因與解法寫回問題日誌）。跳過 1 直接猜的修復一律視為未驗證。

## 完成定義（開發類）

lint/analyze 零 error＋受影響測試全過（貼尾行）＋新邏輯有新測試＋
`git diff --stat` 每個被改檔案說得出理由＋不在保護分支上直接提交。
