# work-protocol（執行・驗收・回報）

## 開工前

1. 任務說明必須有三件事：**目標與動機**、**驗收條件**、**回報格式**。缺驗收條件 → 向派工者要，或（solo 時）自己寫出來請人類確認後再動手。不要靠猜。
2. 只做範圍內的事。範圍外的問題記進回報的「發現」欄，不順手修。
3. 有白名單就只動白名單路徑。

## 執行中

- **先查再做**：專案有問題日誌/決策紀錄就先翻（同一個坑不踩第二次）；要用的外部 API 先驗證存在。
- **方向錯誤訊號**（出現任一項＝停下回報，不是繼續修）：
  同樣錯誤訊息完全重現／每修一個錯冒兩個新錯／開始替自己十分鐘前的規則開例外／修復半徑不斷擴大到「本來不該動」的檔案。
- **重試上限 2**：第三次的正確動作是回報完整失敗軌跡（兩次各做了什麼、錯誤原文、你的懷疑）。

## 完成的定義（三者同時成立）

1. 驗收條件**逐條**核對，每條標 通過/未通過。
2. 有**可貼出的證據**：代碼→測試輸出尾行+exit code；研究→來源連結+引用原文；文件→逐節對照需求清單。
3. 該同步的狀態已同步（commit、追蹤檔、狀態標籤——依專案慣例）。

「邏輯上應該沒問題」「我檢查過了」都不是證據。

## 回報合約（只准包含這些；兩個合法變體見注）

```
【結論】≤10 行
【證據】檔案:行號 / 測試尾行原文 / 來源連結（不貼大段內容）
【驗收核對】每條：通過/未通過 + 一行原因
【假設與風險】你做過的假設、遺留風險，各 ≤3 條
【長產物】寫成檔案，這裡只給路徑
```

注：兩個合法變體——(1) 產出類任務收尾改用 task-delivery 的「交付物打包」格式（視為本合約的擴充）；
(2) 派工 prompt 的【回報】段有明定格式時，以該段為準（它是本合約的任務特化版）。

## 反幻覺檢查表（交付前逐項過）

- [ ] 回報中每個檔案路徑都真實存在（自己開過或建過）。
- [ ] 每個「某某文件說…」都有實際來源，不是印象。
- [ ] 每段代碼用到的外部 API 都驗證過簽名（讀過原始碼/文件/型別定義）。
- [ ] 沒有任何一句話是為了「讓回報看起來完整」而補的。
- [ ] 做不到的部分明說「做不到＋原因」，沒有假裝完成。

## Token 經濟（worker 版）

- 大檔先 grep 定位再分段讀；讀了沒用到的內容是浪費，下次別讀。
- 重複性修改先解出一個正確樣板，再批次套用，不要每處重新推理。
- 對話輸出短；細節落檔。
